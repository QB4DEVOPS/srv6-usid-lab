What Part 3 Proves

Part 1 proved that pinging fcbb:bb00:7:: (PE7’s uN SID) works and shows the SID in the PCAP destination. But the path was just the IGP shortest path — ECMP between P3 and P4. You didn’t control which one was used, and you saw a single SID in the packet. That’s uSID reachability, not uSID compression.

Part 3 has two goals:

  1. Part A — Make compression visible in a packet. By pinging fcbb:bb00:4:7:: (two node IDs packed into one 128-bit address), you force the path via P4 and produce a PCAP that literally shows the compression. P4’s PCAP shows the pointer has shifted. That is the click moment for uSID.
  2. Part B — Automate it with SR-TE. An SR-TE policy on P1 with an explicit segment list does the same thing automatically for any traffic the policy steers. This is how operators use uSID compression in production.
Why Part 3 comes before Part 2 is not a prerequisite

SR-TE is a transport/underlay capability. It works on top of the uSID IS-IS underlay alone — no VRFs, no BGP, no CEs required. Part 2 (L3VPN) and Part 3 (SR-TE) are parallel extensions of Part 1, not a linear chain. You can do them in either order.

How uSID Compression Works

With the F3216 format (32-bit block + 16-bit node IDs), a segment list of N hops is compressed into a single 128-bit IPv6 address by packing each node’s 16-bit ID into successive 16-bit slots after the 32-bit block.

F3216 uSID address structure:

|<--- 32-bit block --->|<-- 16-bit -->|<-- 16-bit -->|<-- 16-bit -->| ...
  fcbb:bb00              node 1 ID       node 2 ID       node 3 ID

2-hop list [P4, PE7]:
  fcbb:bb00 : 0004 : 0007 : 0000 : 0000 : 0000 : 0000 : 0000
           = fcbb:bb00:4:7::  (compressed)

When P4 receives this packet:
  1. Matches its locator fcbb:bb00:0004::/48
  2. uNode behavior: shift left by 16 bits, zero the vacated slot
  3. New destination: fcbb:bb00:0007:0000:... = fcbb:bb00:7::
  4. Forward toward PE7

PCAP comparison:
  P1 → P4 link:  dst = fcbb:bb00:4:7::   (2 hops, compressed)
  P4 → PE7 link: dst = fcbb:bb00:7::     (1 hop, pointer shifted)

The two-PCAP comparison is the only way to see this. Reading about it is not the same.

Address Plan & the Compressed SID

Same topology and addressing as Part 1. The only new address is the 2-hop compressed SID:

AddressMeaningUsed in
fc00:0:7::1PE7 IS-IS loopback (reachability)Part 1 “before” ping
fcbb:bb00:7::PE7 uN SID (single-hop)Part 1 “after” ping
fcbb:bb00:4:7::2-hop compressed SID: P4 then PE7Part 3 compression proof
fcbb:bb00:4::P4 uN SID (single-hop to P4)SR-TE segment list index 10
How fcbb:bb00:4:7:: is constructed

Block fcbb:bb00 + P4 node ID 0004 + PE7 node ID 0007 = fcbb:bb00:0004:0007::. IPv6 drops leading zeros in each group: fcbb:bb00:4:7::. This address lives inside P4’s locator block fcbb:bb00:0004::/48, which is why P4 processes it — P4 owns the /48 that this address falls under.

Part A — Manual uSID Compression Proof

No new configuration needed for Part A. It works with the Part 1 underlay as-is — the uSID data plane already understands compressed multi-hop addresses. You just need to ping the right address and have PCAPs running on the right links.

Step 1 — Verify ECMP Baseline

First confirm that without any steering, traffic from PE5 to PE7 takes an ECMP path via either P3 or P4. Run the traceroute several times — you should see both P3 and P4 appear.

PE5# traceroute ipv6 fcbb:bb00:7:: source fc00:0:5::1

Expected (ECMP — you may see P3 or P4 in hop 2):

PE5# traceroute ipv6 fcbb:bb00:7:: source fc00:0:5::1
Type escape sequence to abort.
Tracing the route to FCBB:BB00:7::
  1 2001:15::1 1 msec 1 msec 1 msec
  2 2001:37::7 2 msec          <-- via P3 (2001:37::3 is P3's link to PE7)
    2001:47::7 2 msec 2 msec   <-- via P4 (2001:47::4 is P4's link to PE7)

✓ Pass: ECMP confirmed — both P3 and P4 paths visible

Step 2 — Arm PCAP on Both Links

You need captures on two links to see both sides of the pointer shift:

  • Link l2 (P1–P4, CML link): will show dst = fcbb:bb00:4:7::
  • Link l13 (P4–PE7, CML link): will show dst = fcbb:bb00:7::

In CML Workbench: right-click each link → Packet Capture → Start. Run captures on both simultaneously before sending the test packet.

Which CML link is which?

P1–P4 is the link between P1 GigabitEthernet0/0/0/2 and P4 GigabitEthernet0/0/0/2. P4–PE7 is the link between P4 GigabitEthernet0/0/0/4 and PE7 GigabitEthernet2. Check the CML topology canvas — hover over links to confirm.

Step 3 — Ping the Compressed SID from PE5

Send the ping using the 2-hop compressed SID as the destination. This will force the path via P4 to PE7 — not P3.

PE5# ping ipv6 fcbb:bb00:4:7:: source fc00:0:5::1 repeat 10

Expected:

PE5# ping ipv6 fcbb:bb00:4:7:: source fc00:0:5::1 repeat 10
Type escape sequence to abort.
Sending 10, 100-byte ICMP Echos to FCBB:BB00:4:7::, timeout is 2 seconds:
Packet sent with a source address of FC00:0:5::1
!!!!!!!!!!
Success rate is 100 percent (10/10), round-trip min/avg/max = 1/2/5 ms

✓ Pass: 10/10 — packet reached PE7 via the forced P4 path

Confirm the path was forced via P4 (not ECMP) with a traceroute:

PE5# traceroute ipv6 fcbb:bb00:4:7:: source fc00:0:5::1

Expected (P4 only — no P3):

PE5# traceroute ipv6 fcbb:bb00:4:7:: source fc00:0:5::1
  1 2001:15::1 1 msec 1 msec 1 msec
  2 2001:47::7 2 msec 2 msec 2 msec   <-- P4's link to PE7, NOT P3

✓ Pass: only P4-side hop appears — P3 path eliminated by explicit SID

Step 4 — Compare PCAPs: Compression then Pointer Shift

Stop both PCAPs and examine the ICMPv6 Echo packets in Wireshark.

P1→P4 link PCAP — the compressed SID

Look at the IPv6 destination field of the ICMP Echo requests:

Frame 1: IPv6 Source: FC00:0:5::1 Destination: FCBB:BB00:4:7:: ← TWO hops compressed into one address Next Header: ICMPv6 (58) Payload: ICMP Echo Request

The destination FCBB:BB00:4:7:: encodes both P4 (node 4) and PE7 (node 7) in a single 128-bit IPv6 address. No SRH extension header is needed — the entire segment list is in the destination field itself. This is the efficiency gain uSID compression delivers over classic SRv6 with SRH.

P4→PE7 link PCAP — the pointer shift

The same packets, one hop later:

Frame 1: IPv6 Source: FC00:0:5::1 Destination: FCBB:BB00:7:: ← P4 shifted the pointer; PE7's SID is now the destination Next Header: ICMPv6 (58) Payload: ICMP Echo Request

P4 processed the uSID: it recognised fcbb:bb00:0004::/48 as its own locator, read the next 16-bit slot (0007 = PE7’s node ID), shifted the destination left by 16 bits, and forwarded toward PE7. The ICMPv6 payload is untouched throughout.

What you just proved

Two PCAPs, one packet, two different destination addresses. The difference is P4’s uSID micro-segment processing. The segment list [P4→PE7] was encoded in one 128-bit address, consumed one hop at a time, with no additional headers. That is what “micro-SID compression” means in practice.

✓ Part A complete: uSID compression proven in a packet

Part B — SR-TE Automation

Part A used a manual ping to an explicit compressed SID. In production, operators use SR-TE policies to automate this: a policy encodes the segment list, and traffic is steered into the policy based on BGP color communities or explicit FIB entries. The resulting packet on the wire is identical to Part A — the compressed SID appears as the IPv6 destination — but the operator never manually computes or types fcbb:bb00:4:7::. The control plane generates it.

Step 5 — Configure SR-TE on P1

The solution YAML already has this configured on P1. If you are working from a fresh Part 1 baseline, paste these two blocks onto P1 separately (segment-list first, policy second — IOS XRv may reject the policy if the segment-list is not yet committed).

Block 1 — Segment list (paste and commit first)

segment-routing
 traffic-eng
  segment-lists
   srv6
    sid-format usid-f3216
   segment-list via-p4-to-pe7
    srv6
     index 10 sid fcbb:bb00:0004::
     index 20 sid fcbb:bb00:0007::
!

Block 2 — Policy (paste after segment-list is committed)

segment-routing
 traffic-eng
  policy to-pe7-via-p4
   color 100 end-point ipv6 fcbb:bb00:0007::
   srv6
    locator MAIN
   candidate-paths
    preference 100
     explicit segment-list via-p4-to-pe7
!
IOS-XE note

cat8000v 17.18 (PE5–PE8) does not accept SR-TE policy configuration. P1–P4 (IOS XRv 9000) are the head-ends for this lab. In a production network with IOS-XE as PE head-end, a later IOS-XE release or XRd would be used.

Note the Binding SID allocated from P1’s locator. This is the local SID that other nodes can push to steer traffic through this policy.

✓ Pass: Policy Up, Oper Up, candidate-path valid

6.2 — SR-TE forwarding entry shows the compressed SID

Expected on P1:

RP/0/RP0/CPU0:P1# show segment-routing traffic-eng forwarding policy name to-pe7-via-p4

Policy name: to-pe7-via-p4   Color: 100   Endpoint: FCBB:BB00:7::
  Outgoing interface: GigabitEthernet0/0/0/2   Next-hop: 2001:14::4
  Outgoing label stack: {FCBB:BB00:4:7::}      <-- compressed SID in the FIB

The FIB entry for this policy shows FCBB:BB00:4:7:: as the outgoing “label” (it is an IPv6 destination, not an MPLS label). This is exactly the same compressed SID you saw in the PCAP in Part A.

✓ Pass: FIB shows compressed 2-hop SID FCBB:BB00:4:7::

Step 7 — Break-Fix: Fail the Forced Path

Shut down the P1–P4 link (P1 GigabitEthernet0/0/0/2) and observe what happens to the SR-TE policy.

RP/0/RP0/CPU0:P1# conf t
interface GigabitEthernet0/0/0/2
 shutdown
commit

Expected — policy goes down immediately:

RP/0/RP0/CPU0:P1# show segment-routing traffic-eng policy name to-pe7-via-p4
  Status:
    Admin: up  Oper: down
  Candidate-paths:
    Preference: 100
      Explicit: segment-list via-p4-to-pe7 (invalid — next-hop unreachable)

Now ping the compressed SID from PE5 again:

PE5# ping ipv6 fcbb:bb00:4:7:: source fc00:0:5::1

Expected (fails — P4 is unreachable from P1):

PE5# ping ipv6 fcbb:bb00:4:7:: source fc00:0:5::1
.....
Success rate is 0 percent (0/5)

But the regular uN SID ping still works (via P3 now):

PE5# ping ipv6 fcbb:bb00:7:: source fc00:0:5::1

✓ Pass: explicit-path ping fails; IGP-path ping succeeds — confirms the forced path was doing real steering

Restore the link:

RP/0/RP0/CPU0:P1# conf t
interface GigabitEthernet0/0/0/2
 no shutdown
commit

✓ Pass: policy returns to Up; both pings succeed again

Troubleshooting

Platform & CML

  • XR cold boot — P1–P4 take ~15–20 min each on cold start; PE cat8000v ~2 min. Wait for all XR nodes = BOOTED before verifying. See Pre-flight checks.
  • Startup config not loaded — P1 at prompt but show running-config hostname shows ios (no IS-IS). Stop → wipe → restart, or verify config in CML before boot.
  • PE locator CLI — show segment-routing srv6 locators fails on cat8000v. Use show isis ipv6 route fcbb:bb00:N:: on PE nodes instead.

Ping to fcbb:bb00:4:7:: fails from PE5

  • Verify IS-IS has a route for fcbb:bb00:0004::/48 on PE5: show isis ipv6 route fcbb:bb00:4::
  • Confirm P4 has micro-segment behavior unode psp-usd in its locator config. Without this, P4 will not process the uSID shift and will drop the packet.
  • Check P4's CEF: show cef ipv6 fcbb:bb00:4:7:: on P4 — should show a forwarding entry toward PE7 after the uSID shift.

SR-TE policy stays in Oper: down

  • Confirm the segment-list was committed first as a separate operation before the policy. IOS XRv validates the segment-list binding before accepting the policy.
  • Check that sid-format usid-f3216 is present under the srv6 stanza of segment-lists — without it the SIDs are interpreted as classic SRv6 addresses, not uSID.
  • Run show segment-routing traffic-eng topology on P1 to confirm P4 and PE7 are visible in the SR-TE topology (both must be reachable via IS-IS).

PCAP shows fcbb:bb00:7:: on P1→P4 (not the compressed SID)

  • This means PE5 is sourcing the ping as a single-hop uSID (not the 2-hop compressed version). Confirm the ping destination is literally fcbb:bb00:4:7::, not fcbb:bb00:7::.
  • IOS-XE may abbreviate the address differently. Try: ping ipv6 FCBB:BB00:0004:0007:: source fc00:0:5::1 (fully expanded).

Traceroute shows P3 instead of P4 despite pinging fcbb:bb00:4:7::

  • The packet is reaching P4’s locator fcbb:bb00:0004::/48, which IS-IS routes to P4. If the traceroute shows P3 as a transit hop, check that P1’s IS-IS route for fcbb:bb00:0004::/48 exits via Gi0/0/0/2 (the P1→P4 direct link), not via P3 as an intermediate.
