Part 3 — SR-TE uSID Traffic Engineering

This lab guide is under development. It will cover explicit path steering using SRv6 uSID segment lists, color-based ODN policies, and verifying steered traffic with PCAP on the same Quisted 8-router topology.

Part 1 already includes an optional SR-TE preview — see the SR-TE uSID policy section in the Part 1 guide for a working single-policy example while Part 3 is being written.

What Part 3 Will Cover

After Parts 1 and 2 you have:

  • An IPv6-only IS-IS underlay with SRv6 uSID locators on all 8 nodes
  • L3VPN services using End.DT4 SIDs carried over the shortest IGP path

Part 3 adds explicit path control — steering VPN or raw traffic along a specific sequence of nodes regardless of the IGP shortest path. That is SR-TE (Traffic Engineering) with uSID segment lists.

Planned lab content

  • Step 1 — Explicit segment list: configure a uSID segment list forcing traffic via P4 then PE7 instead of the ECMP default
  • Step 2 — SR-TE policy: bind the segment list to a color + endpoint, verify with show segment-routing traffic-eng policy
  • Step 3 — Steer VPN traffic: attach an ODN color to a VPNv4 route; confirm PE5 encapsulates with the explicit SID stack in the PCAP
  • Step 4 — Proof run: traceroute from PE5 showing the forced path (not the ECMP path); before/after PCAP comparison showing segment list in the IPv6 destination
  • Step 5 — Failure test: bring down a link in the forced path; confirm the policy falls back or goes down, traffic reverts to IGP path

Key concepts this lab will make concrete

  • uSID segment compression: multiple hops encoded in one 128-bit IPv6 address — the feature that makes SRv6 uSID efficient compared to classic SRH stacks
  • Binding SID: a single local SID that represents an entire SR-TE policy — other PEs just push the binding SID
  • ODN (On-Demand Next-hop): dynamic policy instantiation triggered by BGP color extended community — no manual policy per-prefix
  • Platform differences: IOS-XR traffic-eng stanza vs IOS-XE equivalent for the head-end PE

Preview — Try It Now in Part 1

The Optional: SR-TE uSID policy section in Part 1 already contains a working single-policy example on P1 (IOS-XR). It forces traffic from PE5 via P4 to PE7 using an explicit uSID segment list. That section is a preview of Part 3 — the full lab will extend it to IOS-XE head-ends, L3VPN color steering, and failure scenarios.
