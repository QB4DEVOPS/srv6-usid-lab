What’s New in Part 4

Part 2 simulated customer networks using Loopback100 interfaces on the PE routers. Part 4 replaces that with real CE routers (IOSv) and introduces two important PE-CE routing patterns side-by-side:

Left side — eBGP CE-PE (CE9, CE10)

CE9 and CE10 are dual-homed — each has two uplinks, one to PE5 and one to PE6. They run eBGP to both PEs and advertise their customer prefix. This is the modern, production-grade PE-CE routing method.

Right side — static / non-speaker (CE11, CE12)

CE11 and CE12 have no routing protocol. Each CE has a static default route pointing at its primary PE. The PE has a static route for the CE’s prefix inside the VRF, redistributed into BGP VPNv4 with redistribute static. The CE is a “non-speaker” — it never participates in any routing protocol.

Both approaches deliver the same SRv6 VPN transport — the CE has no knowledge of SRv6 in either case. The difference is entirely at the PE-CE boundary.

VRF model change: In Part 2, each PE carried only one VRF. In Part 4, all four PEs carry both CUST-A and CUST-B so that dual-homed CEs can reach PEs on both sides of the network via a single VRF.

Step 1 — Verify VRFs and CE-Facing Interfaces on All PEs

Each PE must have both CUST-A and CUST-B defined, and both Gi4 and Gi5 up/up in the correct VRF.

Check on PE5 (representative — run on all four PEs)

Expected:

PE5# show vrf detail | include ^VRF|State
VRF CUST-A (VRF Id = 1); default RD 65000:100; default VPNID <not set>
  State: Up
VRF CUST-B (VRF Id = 2); default RD 65000:200; default VPNID <not set>
  State: Up

PE5# show ip interface brief | include GigabitEthernet[45]
GigabitEthernet4          10.9.5.2        YES manual up       up
GigabitEthernet5          10.10.5.2       YES manual up       up

✓ Pass: both VRFs Up, both CE-facing interfaces up/up on PE5

✓ Pass: same result on PE6 (10.9.6.x / 10.10.6.x), PE7 (10.11.7.x / 10.12.7.x), PE8 (10.11.8.x / 10.12.8.x)

CE9 and CE10 each maintain two eBGP sessions — one to PE5 and one to PE6. After both PEs come up, the CE should have two active paths to the VPN.

Expected on CE9:

CE9# show ip bgp summary

10.9.5.2        4        65000      12      12        3    0    0 00:01:10          1
10.9.6.2        4        65000      12      12        3    0    0 00:01:08          1

2.2 — CE9 routing table shows paths to the VPN via both PEs

Step 3 — Verify Non-Speaker CEs (Right Side)

CE11 and CE12 have no routing protocol. They only have static default routes pointing to their primary PE. The PE holds static routes for the CE prefix inside the VRF.

3.1 — CE11 has no BGP, no dynamic routing — only static default

Expected on CE11:

CE11# show ip route

Gateway of last resort is 10.11.7.2 to network 0.0.0.0

S*    0.0.0.0/0 [1/0] via 10.11.7.2
      10.0.0.0/8 is variably subnetted
C        10.11.0.1/32 is directly connected, Loopback0
C        10.11.7.0/30 is directly connected, GigabitEthernet0/0
C        10.11.8.0/30 is directly connected, GigabitEthernet0/1

No BGP entries. CE11 just has a default route via PE7 and a backup via PE8 (metric 10). The CE is completely unaware of the SRv6 network or VPN topology.

✓ Pass: CE11 has only static default, no routing protocol

3.2 — PE7 has static route for CE11 in VRF CUST-A and redistributes it

Expected on PE7:

PE7# show ip route vrf CUST-A

      10.0.0.0/8 is variably subnetted
S        10.11.0.1/32 [1/0] via 10.11.7.1       <-- static for CE11
C        10.11.7.0/30 is directly connected, GigabitEthernet4
B        10.9.0.1/32 [200/0] via FC00:0:5::1 (SRv6), 00:03:12   <-- CE9 from SRv6 domain
      192.168.70.0/24 is variably subnetted
C        192.168.70.0/24 is directly connected, Loopback100

The static route for CE11 (10.11.0.1/32) is present and will be redistributed into VPNv4 by redistribute static in the BGP VRF AF. CE9’s prefix (10.9.0.1/32) arrived via SRv6 VPNv4 from PE5.

✓ Pass: static for CE11 present; CE9’s VPN route installed via SRv6

3.3 — CE11’s prefix visible in VPNv4 on PE5

Expected on PE5 (CE11’s prefix arrived from PE7 via SRv6 VPNv4):

PE5# show bgp vpnv4 unicast vrf CUST-A

          Network          Next Hop            Metric LocPrf Weight Path
 *>i  10.11.0.1/32      FC00:0:7::1              0    100      0 ?
 *>   10.9.0.1/32       10.9.5.1                 0             0 65100 i
 *>   192.168.50.0/24   0.0.0.0                  0         32768 ?

The non-speaker CE11’s prefix (10.11.0.1/32) arrived from PE7 (FC00:0:7::1) with no AS path — because there is no BGP on the CE side, it was redistributed as a static route and appears as an internal (?) origin.

✓ Pass: CE11’s prefix visible at PE5 via SRv6 VPNv4 from PE7

Step 4 — Verify VPN Route Propagation

With all routes installed, confirm the full routing table is correct from the CE perspective on both sides.

4.1 — CE9 receives CE11’s prefix via eBGP from PE5

Expected on CE9:

CE9# show ip bgp

          Network          Next Hop            Metric LocPrf Weight Path
 *>   10.9.0.1/32       0.0.0.0                           32768 i
 *>i  10.11.0.1/32      10.9.5.2                 0             0 65000 ?
 *>   192.168.50.0/24   10.9.5.2                 0             0 65000 ?

CE9 receives 10.11.0.1/32 (CE11’s non-speaker prefix) from PE5 via eBGP. CE9 has a complete route to CE11 without knowing anything about SRv6 or the static-route pattern PE7 uses.

✓ Pass: CE9 BGP table includes CE11’s prefix via PE5

4.2 — CE11 can reach CE9’s prefix via its static default

CE11’s default route points to PE7 (10.11.7.2). PE7’s CUST-A VRF has 10.9.0.1/32 via FC00:0:5::1 (SRv6 VPNv4 from PE5/PE6). Traffic from CE11 destined for 10.9.0.1 flows: CE11 → PE7 (VRF lookup) → SRv6 encap → P3/P4 → P1 → PE5 (End.DT4 decap) → CE9.

5.4 — Dual-homed failover: disconnect CE9–PE5 link, ping must still pass via PE6

In CML, use the link breakpoint tool to bring down the CE9–PE5 link (l14). CE9’s BGP session to PE5 drops; CE9 re-routes through PE6 within the BGP hold-down period (~90s default). Pings via source 10.9.0.1 should recover without any CE11 config change.

✓ Pass: CE-to-VPN reachability survives single uplink failure (dual-homed value demonstrated)

Troubleshooting

CE9 eBGP session stays Active / Idle

  • The session is IPv4 between CE9’s GigabitEthernet0/0 address and PE5’s GigabitEthernet4 address (both in CUST-A VRF). Check that both interfaces are up: show ip interface brief | include GigabitEthernet4 on PE5.
  • The eBGP session does not require update-source; it uses the directly connected link address. If the link is up/up and the session is still Active, check that neighbor 10.9.5.1 remote-as 65100 is present under address-family ipv4 vrf CUST-A on PE5.

CE11’s prefix not appearing in VPNv4 at remote PEs

  • Check that redistribute static is present under address-family ipv4 vrf CUST-A in PE7’s BGP config (and PE8 for redundancy).
  • Confirm the static route itself is installed: show ip route vrf CUST-A 10.11.0.1 on PE7 must show S. If PE7 cannot ping 10.11.7.1 from the VRF, the CE11 interface may not be up.

CE9 does not receive CE11’s prefix from PE5/PE6

  • PE5 must have the route in its VRF table before it can advertise it to CE9 via eBGP. Check show bgp vpnv4 unicast vrf CUST-A on PE5 for 10.11.0.1/32. If absent, PE7 has not redistributed it yet.
  • Verify send-community extended is configured on all VPNv4 iBGP neighbors — without this, the Route Target extended community is stripped and the remote PE will not import the route.

CE9 pings succeed but CE11 pings CE9 fail

  • CE11’s default route (ip route 0.0.0.0 0.0.0.0 10.11.7.2) must point to PE7’s VRF-facing address. Check that PE7’s GigabitEthernet4 is up and has 10.11.7.2/30 in VRF CUST-A.
  • If CE11 pings PE7’s Gi4 address (10.11.7.2) but cannot reach CE9’s Lo0, the issue is in the VPN routing, not the CE-PE link.

Dual-homed failover is very slow

  • Default BGP hold-time is 180s, keepalive 60s. A link failure triggers hold-down expiry in ~90–180s unless BFD is configured. For the failover demonstration in Step 5.4, wait the full hold-down period or configure neighbor X.X.X.X fall-over on both CE and PE to detect the failure faster.

Appendix — CE Router Configs

CE routers are IOSv. No SRv6 configuration anywhere — they are standard IP routers.

CE9 (CUST-A, eBGP AS 65100, dual-homed to PE5+PE6)

hostname CE9
no ip domain lookup
ip routing
!
interface Loopback0
 ip address 10.9.0.1 255.255.255.255
!
interface GigabitEthernet0/0
 description to PE5 (CUST-A primary)
 ip address 10.9.5.1 255.255.255.252
 no shutdown
!
interface GigabitEthernet0/1
 description to PE6 (CUST-A secondary)
 ip address 10.9.6.1 255.255.255.252
 no shutdown
!
router bgp 65100
 bgp router-id 10.9.0.1
 bgp log-neighbor-changes
 neighbor 10.9.5.2 remote-as 65000
 neighbor 10.9.6.2 remote-as 65000
 !
 address-family ipv4
  network 10.9.0.1 mask 255.255.255.255
  neighbor 10.9.5.2 activate
  neighbor 10.9.6.2 activate
 exit-address-family
!
