What is SRv6 L3VPN?

In MPLS L3VPN, the PE router pushes a VPN label (from an LDP/BGP label binding) plus a transport label (from MPLS-SR or LDP) onto every customer packet. Two label operations per hop, two control planes to maintain.

SRv6 L3VPN removes this duality. The SRv6 VPN SID—a single IPv6 address allocated from the egress PE’s locator block—encodes both “deliver to this PE” and “look up in this VRF” in one field. The core (P1–P4) forwards the outer IPv6 packet normally using IS-IS routes; no label stack required.

End.DT4 — the VPN SID behavior

The SID behavior assigned to a VPN SID is End.DT4 (Endpoint with Decapsulation and Table lookup, IPv4). When PE7 receives an outer IPv6 packet addressed to its End.DT4 SID:

  1. The outer IPv6 header is stripped (decapsulated).
  2. The inner IPv4 packet is looked up in the VRF bound to that SID (e.g., CUST-A).
  3. The packet is forwarded out the appropriate VRF interface.

No MPLS label operations, no penultimate-hop popping decisions, no RSVP or LDP sessions needed.

SRv6 L3VPN vs MPLS L3VPN — control-plane comparison
Feature MPLS L3VPN SRv6 L3VPN
VPN service identifierVPN label (allocated per-prefix or per-VRF)SRv6 SID (IPv6 address from PE locator)
TransportMPLS label stack (transport + VPN label)Single outer IPv6 header; SID = transport + VPN
P-router awareness of VPNNo (label swap only)No (IPv6 forwarding only)
BGP address-familyVPNv4 / VPNv6 with MPLS next-hopVPNv4 / VPNv6 with SRv6 SID as service label

SRv6 Interworking Gateway (IGW)

In this lab all four PE routers run SRv6, so the VPN SID travels end-to-end without translation. In a real network where a remote PE does not support SRv6 (e.g., it only speaks SR-MPLS), a border device called an SRv6 Interworking Gateway (IGW) is needed. The IGW translates the SRv6 VPN SID into an MPLS VPN label (or vice versa) at the domain boundary—it is the SRv6 equivalent of an inter-AS Option B ASBR, but for the data-plane translation. This lab does not include an IGW because all PEs are SRv6-capable.

BGP router-id in an IPv6-only environment

IOS-XE requires an IPv4 BGP router-id (dotted-decimal format), but bgp router-id X.X.X.X accepts any arbitrary IPv4 value—it does not need to be configured on any interface. No IPv4 address needs to be added to Loopback0. The router-id is purely a BGP identifier sent in OPEN messages; it never needs to be reachable.

IOS-XE version note

The commands segment-routing srv6, locator MAIN, and alloc mode per-vrf under BGP address-families were introduced progressively across IOS-XE 17.x releases. On cat8000v 17.18, these are expected to be available. If IOS-XE rejects segment-routing srv6 under address-family vpnv4 unicast, try placing the segment-routing srv6 locator MAIN command directly under address-family ipv4 vrf CUST-A instead, without the global VPNv4-level block.

Troubleshooting / Gotchas

BGP session stays Active / Idle

  • Root cause: IS-IS does not have a route for the neighbor’s IPv6 loopback, or the SRv6 locator is not Up.
  • Verify: ping fc00:0:7::1 source fc00:0:5::1 from PE5. If this fails, go back to Part 1 Steps 3–4.
  • Check: show ipv6 route fc00:0:7::1/128 on PE5 must show an IS-IS route.
  • Check: update-source Loopback0 is configured — BGP must source from fc00:0:5::1, which IS-IS knows about.

BGP session up but no VPNv4 routes received

  • Check send-community extended is configured on all neighbors — extended communities carry the Route Target.
  • Check that both ends have the same RD/RT values. A typo in the RT means the importing PE will not install the route.
  • Check: show bgp vpnv4 unicast all neighbors fc00:0:7::1 received-routes — routes arriving but not installed may indicate an RT mismatch.

End.DT4 SID does not appear in show segment-routing srv6 sid

  • Step 8 (Block C) may not be fully applied. Check show running-config | section router bgp and confirm segment-routing srv6 and alloc mode per-vrf are present under address-family ipv4 vrf CUST-A.
  • If segment-routing srv6 under address-family vpnv4 unicast is not accepted by your IOS-XE version, omit that block and only keep the per-VRF stanza; SID allocation still works from the per-VRF binding alone.
  • Confirm the locator name exactly matches: locator MAIN (case-sensitive on IOS-XE).

VRF route installed but ping fails

  • Check CEF: show ip cef vrf CUST-A 192.168.70.0/24 detail. The SRv6 headend entry must show the remote End.DT4 SID.
  • If CEF shows the route but ping fails, confirm the P-router IS-IS route for fcbb:bb00:0007::/48 is present: show ipv6 route fcbb:bb00:7:: on P1.
  • If CEF is missing entirely after BGP installs the route, try clear ip bgp * soft or clear ip cef vrf CUST-A to force re-resolution.

Traceroute shows * * * for intermediate hops

This is normal for SRv6 transit. P routers may not reply to TTL-exceeded messages in the SRv6 outer header depending on platform configuration. The absence of intermediate hop replies does not indicate a forwarding fault as long as the destination ping succeeds.

IOS-XE BGP router-id warning

If you see %BGP-4-NOTIFRCV: VRF default: Neighbor ... sent NOTIFICATION after adding the BGP config, check that bgp router-id X.X.X.X is explicitly present in your router bgp 65000 stanza. If it is omitted, IOS-XE tries to auto-select a router-id from a locally configured IPv4 address; since Loopback0 is IPv6-only in this lab, no candidate exists and BGP will fail to start.
